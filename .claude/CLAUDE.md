## 删减需显式授权

判据看对外行为/契约是否改变——删代码连带删了行为就等于删功能。会改变它的删除或变更(功能、接口、数据字段、配置、默认值、文案、承载行为的代码路径),不在本次要求内默认先问。"更优 / 冗余 / 顺手清理"不是授权;典型越权:"移除 X 改 Y"而 X 不在要求内。

例外:本次改动自产的孤儿 import / 变量 / 函数可清;既存死代码仍先问。

## 第一性原理

- 下性能/根因/行为结论前要有可验证证据(运行时真值、测试、度量);没证据按假设处理,先问或先验。约束只压在会驱动改动/决策的结论,不是每个中间推理句。
- 注释和文档不是权威,结合实际代码验证。
- 别拿惯例 / 训练先验当依据(「业界都这样 / 一直这么写」不算),落到本项目的真实需求与约束。
- 复杂度只会转移不会消失;评估方案看它把复杂度放对地方没有,不看是否"看起来整洁"。
